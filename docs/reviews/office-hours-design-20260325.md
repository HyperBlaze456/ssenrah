## Problem Statement

The current agent philosophy (`docs/harness_architecutre/initial_philosophy.md`) is a comprehensive literature review — 8 pillars, 40+ requirements across 9 domains — but it's structured as a flat list of equal priorities derived from external research (Deloitte, Karpathy, CoSAI, etc.). It doesn't reflect the founder's own pain or establish a dependency hierarchy between capabilities. The result is a spec that tries to serve everyone and therefore serves no one.

The founder, a prompt engineer, identified two concrete pain points:
1. **Agent coordination is broken.** Teams can only cooperate at a single depth. Status gets lost between agents. Prerequisites aren't signaled.
2. **Tool calling destroys context quality.** Agents waste tokens fumbling for the right tool, degrading the signal-to-noise ratio in the context window.

The status quo (Claude Code + oh-my-claudecode plugin) "works but works too much" — brute-forcing through problems with no learning across compaction boundaries.

## Demand Evidence

The founder's own pain as a prompt engineer running multi-agent workflows. Not yet validated with external users, but the pain is specific and reproducible: agents losing coordination state, tool call waste, and compaction-induced amnesia ("Groundhog Day for agents"). Builder.io's "AI Agent Orchestration is Broken" publication and GitHub Engineering's multi-agent failure analysis independently confirm the same pain points.

## Status Quo

Claude Code with oh-my-claudecode plugin. It gets tasks done but inefficiently — exploring every wrong path, no active learning, no streamlined flow. After context compaction, agents must rediscover information they already found. There is no visibility into what agents are doing during long-running tasks (8+ hour runs are black boxes).

## Target User & Narrowest Wedge

**Target user:** The founder — a prompt engineer who values context quality above all else and needs to see, steer, and tune multi-agent work.

**Narrowest wedge:** A transparency layer that makes agent work visible — agents communicating on interfaces where the user can monitor (Slack/Discord/custom), backed by system-level telemetry. No black box, especially when agents coordinate.

**Long-term vision:** An engine that is a "platform that creates platforms" — general enough for enterprise use, but opinionated for the first user.

## Premises

1. **Dependency chain, not flat pillars.** The philosophy must be structured as a layered dependency: transparency → coordination → orchestration → memory → self-improvement, plus cross-cutting concerns that apply across all layers. The dependency is primarily vertical (each layer builds on the one below), but not strictly linear — orchestration and memory have lateral dependencies (typed handoffs need state, state needs structure). Layers 2 and 3 are co-dependent and built together in Tier 2. This differs from the original doc's Part III tiering, which placed Planner-Worker (C-1) in Tier 1; the revised philosophy deliberately defers orchestration until visibility and coordination are solid.

2. **Agents talk naturally, the system records everything.** Agents communicate in natural language (not structured JSON events). The system wraps each agent turn — who sent what to whom, at what time, in response to what task — into structured event log entries automatically. This is metadata wrapping, not NLP parsing; the system records the *fact* of communication, not the *semantic content*. The event log serves telemetry, replay, and debugging. Human-facing interfaces (Slack/Discord/custom UI) are bidirectional projections — humans observe and intervene, and their input flows back as events. Note: the existing harness already logs structured JSONL events and XML-based intent blocks at the system level; this is compatible — agent-to-agent communication is natural language, system-level telemetry remains structured.

3. **Opinionated wedge, open architecture.** The philosophy is opinionated for the first user (a prompt engineer who needs to see and tune agents), but the engine's architecture is a "platform that creates platforms" — concretely: the engine provides composable primitives (task graphs, event buses, policy engines, tool registries, agent type schemas) that downstream users assemble into domain-specific agent platforms. The engine does NOT prescribe what agents do; it prescribes how they coordinate, report, and are observed.

4. **Thesis from pain, not from predictions.** The founder's thesis leads. External citations (Deloitte, Karpathy, CoSAI, Gartner, etc.) serve as supporting evidence, not as the foundation.

## Cross-Model Perspective

Codex independently reviewed the session and challenged Premise 2:

> "Agent communication should not happen on Slack/Discord as the primary substrate. Those are presentation layers. The source of truth should be a typed event log plus materialized run state."

Key Codex contributions:
- **Steelman:** "An event-sourced operating system for agent teams" — canonical shared state, typed handoffs, compaction-safe resume packets.
- **Key signal:** "It's me, honestly" — build a power tool for one expert operator first, not an enterprise suite.
- **Prototype:** Local-first TypeScript, SQLite event store, React Flow graph UI, append-only events, typed handoffs, compaction-safe resume packets.

**Resolution:** The founder agreed that an event log is important for telemetry and system-level tracking, but rejected structured JSON events as the agent communication mechanism. Agents should communicate naturally (language); the system captures and structures communications automatically. Slack/Discord/custom UI are bidirectional projections of the system's event log.

## Approaches Considered

### Approach A: Dependency Chain Manifesto (CHOSEN)
Restructures the 8 pillars into a 5-layer dependency chain. Each layer explains WHY it depends on the one below. Requirements are embedded within each layer, not in a separate section. Leads with founder's thesis, citations as supporting evidence.
- Effort: S | Risk: Low
- Pros: Clean structure, clear priority ordering, actionable
- Cons: May lose some breadth of the original doc

### Approach B: First-Person Thesis + Architecture
Rewrites as a personal thesis ("Here's what I learned as a prompt engineer"). Three sections: Thesis, Architecture, Wedge. Drops requirements tables entirely.
- Effort: S | Risk: Med (loses reference material)
- Pros: Sharp, personal, concise
- Cons: Too thin for downstream architecture decisions

### Approach C: Layered Spec with Original as Appendix
New doc leads with dependency chain. Requirements reorganized by layer, narrowed to ~20. Original preserved as appendix.
- Effort: M | Risk: Low
- Pros: Nothing lost, everything restructured
- Cons: Longer document, may feel academic

## Recommended Approach

**Approach A: Dependency Chain Manifesto.** The revised philosophy should have this structure:

### Proposed Document Structure

```
# Agent Philosophy: Dependency Chain Manifesto

## Thesis (1 page)
- What's broken: the founder's pain in concrete terms
- What this engine is: a platform that creates platforms
- The core insight: transparency enables everything else

## Layer 0: Visibility — See What Agents Do
- Agents communicate naturally; the system wraps turns into structured event log entries
- Event log as system of record (metadata wrapping, not NLP parsing)
- Human-facing projections: Slack/Discord/custom UI (bidirectional — humans observe + intervene)
- Developer dashboard for telemetry deep-dives (traces, costs, anomalies)
- Original requirements mapped here: D-1 (tracing), D-2 (cost tracking), D-3 (reasoning logs), D-4 (anomaly detection), D-5 (test/verify environments), D-6 (progress dashboard)

## Layer 1: Coordination — Agents Signal Each Other
- Task dependency graphs that agents can read and act on
- Status awareness: done / blocked / need input — derived from task graph state changes (e.g., submitResult, claimReadyTasks), not parsed from agent natural language output
- Hierarchical cooperation (not just flat single-depth)
- Original requirements mapped here: C-2 (typed handoffs), C-5 (minimize agent count), C-6 (parallel result synthesis), C-7 (communication protocol)

## Layer 2: Orchestration — Plan, Assign, Verify (co-dependent with Layer 3)
- Planner-Worker separation (C-1)
- Typed handoffs with state contracts (requires memory primitives from Layer 3)
- Graceful degradation (C-4)
- Maker-Checker verification (C-3)
- Workers submit, orchestrator verifies and completes
- Original requirements mapped here: C-1 (planner-worker), C-3 (cross-verification), C-4 (graceful degradation), A-1 (sandboxed execution), A-2 (ephemeral credentials), A-3 (least privilege), A-4 (MCP gateway), A-5 (action whitelist), A-6 (shadow agent detection)

## Layer 3: Memory — State That Survives (co-dependent with Layer 2)
- Compaction-safe resume packets (extends the existing HarnessCheckpoint schema with typed state snapshots that capture tool registry state, partial results, and dependency graph version — not just phase and goal. Lets agents pick up exactly where they left off)
- Three-tier architecture (working/indexed/persistent)
- Active persistence: agents write important info immediately, don't rely on compaction
- Context quality as a first-class metric (prompt engineer's core value)
- Original requirements mapped here: B-1 (3-tier memory), B-2 (async compaction), B-3 (event-based decision logs), B-4 (immediate persistence), B-5 (aggressive compaction for long tasks), B-6 (knowledge graph), B-7 (memory performance monitoring)

## Layer 4: Self-Improvement — Learn From What You Observed
- Requires visibility (Layer 0) to know what happened
- Requires memory (Layer 3) to retain what was learned
- Evaluation pipeline, LLM-as-Judge, closed-loop feedback
- Bounded: can optimize prompts/rules/tools, CANNOT expand own permissions or modify security boundaries
- Original requirements mapped here: E-1 (eval in CI/CD), E-2 (hybrid eval), E-3 (success case storage), E-4 (prod failure → eval dataset), E-5 (circuit breakers), E-6 (self-awareness of limits), E-7 (bounded self-improvement scope)

## Cross-Cutting Concerns
- Cost management and model routing: G-1 (model routing by complexity), G-2 (budget caps at agent/task/org level), G-3 (caching), G-4 (cost-quality tradeoff metrics)
- Existing stack integration via MCP: F-1 (MCP protocol), F-2 (deep context integration), F-3 (SKILL.md packaging), F-4 (legacy API wrapping), F-5 (tool metadata)
- Vendor independence: I-1 (model abstraction), I-2 (OpenTelemetry standard), I-3 (portable config)
- UX patterns: H-1 (fast + slow AI), H-2 (autonomy slider), H-3 (natural language errors), H-4 (fast deployment of improvements)

## Priority
- Tier 1 (build first): Layer 0 (core: event log + CLI/dashboard visibility; stretch: bidirectional Slack/Discord projections) + Layer 1 — visibility and coordination. The event log is the Tier 1 deliverable; projection interfaces are additive. Differs from original doc's Tier 1 which included C-1 (planner-worker); we deliberately defer orchestration until we can SEE what agents do.
- Tier 2 (build next): Layer 2 + Layer 3 (co-dependent, built together) — orchestration and memory
- Tier 3 (build when ready): Layer 4 + cross-cutting concerns
```

## Open Questions

1. **Language of the revised doc:** The original is in Korean. Should the revision stay in Korean or switch to English for broader reach?
2. **How much of the original citations to preserve:** Every layer could reference 3-5 citations. Is that too much supporting evidence, or does it add credibility?
3. **Compaction-safe resume packets:** Codex introduced this concept. How much architectural detail should the philosophy doc include vs. deferring to an architecture doc?
4. **"Platform that creates platforms":** This is the long-term vision. Should the philosophy explicitly scope what the engine does NOT do, or leave that open?
