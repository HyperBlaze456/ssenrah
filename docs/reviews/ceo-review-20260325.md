## Vision

### 10x Check
Instead of just logging and projecting to Slack: a real-time agent operations center. Every AI agent your team runs — Claude Code, Codex, custom agents — appears as an "employee" on a unified dashboard. You see their status, decisions, coordination. You can pause, redirect, inject instructions. The system learns from patterns. **Mission Control for AI agents — the Datadog of agent operations.**

### Platonic Ideal
You open your laptop Monday morning. The dashboard shows: overnight, your 3-agent coding team reviewed 12 PRs, flagged 2 as risky, auto-merged 8, left notes on 2. On Slack, the thread reads like a competent junior dev team reporting progress. You scan in 2 minutes, approve the 2 flagged PRs, and go back to strategic work. It feels like having a team that works while you sleep and tells you what matters when you wake up.

## Strategy: Product-First, Extract Later

The engine is NOT the product. The product is an **Autonomous Dev Team Dashboard** — teams deploy agent teams for coding tasks and watch them work. The engine is the infrastructure underneath.

Key strategic decision: Build the transparency layer first and **test it on existing agent engines** (Claude Code, Codex) rather than building custom agents. This validates the value of visibility without the cost of building orchestration.

## Approach: Minimal Viable Transparency (MVT)

Build ONLY the event log + Slack projection + accepted expansions. No dashboard UI, no coordination layer, no custom agents. Just make existing agents visible on Slack with cost tracking and escalation.

## Accepted Scope (post-Codex scope reduction)

- ONE agent adapter (Claude Code hooks only)
- Event capture → local timeline (SQLite/file-based, append-only)
- Cost tracking per session (from token counts, stored locally)
- Configurable escalation rules (user-defined thresholds, not hardcoded)

## Codex Outside Voice — Key Findings Applied

- Original "MVT" was 5+ features pretending to be minimal. Stripped to true minimum.
- "Not parsing agent language" contradicts stuck detection, threading, and digests — deferred features that need semantics until task identity model is defined.
- Security gap: shipping prompts/code/secrets to external platforms is a data-exfiltration risk — reason to defer Slack projection until security model is designed.
- Cost tracking must be local-only (vendor usage data is delayed/partial).

## Key Constraints

- Philosophy doc (Dependency Chain Manifesto) guides architecture
- Layer 0 (Visibility) is the foundation — build this first
- Agents communicate naturally; system records metadata (not NLP parsing)
- Status derived from task graph API, not agent language parsing
- Pluggable projection interface (Slack today, custom UI later)
