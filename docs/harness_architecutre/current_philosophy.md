# 에이전트 엔진 철학: 의존성 사슬 선언문

> 2026년 3월. 프롬프트 엔지니어의 고통에서 출발하여, 에이전트 하네스가 무엇을 어떤 순서로 해결해야 하는지를 하나의 의존성 사슬로 선언한다.

---

## 테제: 무엇이 망가져 있는가

에이전트 오케스트레이션은 망가져 있다.

이것은 외부 리서치의 결론이 아니다. 매일 멀티에이전트 워크플로우를 돌리는 프롬프트 엔지니어로서 직접 겪는 고통이다:

1. **에이전트 팀의 협업은 단일 깊이에서만 작동한다.** Agent A가 선행 작업을 끝내도 Agent B는 모른다. 상태가 유실된다. 의존성이 신호되지 않는다. 에이전트들은 서로의 존재를 겨우 인식하는 수준에서 "협력"한다.

2. **도구 호출이 컨텍스트 품질을 파괴한다.** 에이전트가 올바른 도구를 찾지 못해 헛되이 시도하고, 실패한 호출이 컨텍스트 윈도우의 신호 대 잡음비를 떨어뜨린다. 토큰은 비싸고, 낭비된 토큰은 단순히 비용이 아니라 에이전트의 판단력 저하로 이어진다.

3. **컨텍스트 압축(compaction) 이후, 에이전트는 모든 것을 잊는다.** 이미 발견한 정보를 다시 찾아야 한다. 이미 시도한 실패를 다시 반복한다. 에이전트의 하루는 *Groundhog Day*다.

현재의 상태(status quo)를 구체적으로 말하면: Claude Code와 플러그인 조합은 작업을 "완수"한다. 하지만 **너무 많이 일한다** — 미로의 모든 막다른 길을 탐색한 뒤에야 출구를 찾는다. 능동적 학습이 없다. 8시간 동안 돌려놓으면 블랙박스다. 무엇을 하고 있는지, 왜 그렇게 하는지 보이지 않는다.

이 고통은 나만의 것이 아니다. Builder.io는 "AI Agent Orchestration is Broken"을 발표했고, GitHub 엔지니어링 팀은 멀티에이전트 워크플로우 실패의 근본 원인이 **커뮤니케이션 프로토콜 문제**임을 확인했다. Gartner는 2027년까지 에이전틱 AI 프로젝트의 40% 이상이 비용 급증과 리스크 관리 실패로 취소될 것이라 경고했다.

---

### 이 엔진이 무엇인가

ssenrah는 **플랫폼을 만드는 플랫폼**이다.

구체적으로: 엔진은 조합 가능한 원시 요소(composable primitives)를 제공한다 — 태스크 그래프, 이벤트 버스, 정책 엔진, 도구 레지스트리, 에이전트 타입 스키마. 다운스트림 사용자는 이 원시 요소들을 조립하여 자신의 도메인에 특화된 에이전트 플랫폼을 구축한다.

엔진은 에이전트가 **무엇을 하는지** 규정하지 않는다. 에이전트가 **어떻게 조율되고, 보고하고, 관찰되는지**를 규정한다.

---

### 핵심 통찰: 의존성 사슬

에이전트 시스템의 능력은 평등한 기둥(pillar)들이 아니다. **의존성 사슬**이다:

```
Layer 0: 가시성 (Visibility)      — 에이전트가 무엇을 하는지 본다
    ↓
Layer 1: 조율 (Coordination)      — 에이전트가 서로 신호한다
    ↓
Layer 2: 오케스트레이션             — 계획하고, 배정하고, 검증한다
    ↔  (상호 의존)
Layer 3: 기억 (Memory)            — 상태가 살아남는다
    ↓
Layer 4: 자기개선 (Self-Improvement) — 관찰한 것으로부터 배운다
```

**각 레이어는 아래 레이어에 의존한다.** 볼 수 없으면 조율할 수 없다. 조율할 수 없으면 오케스트레이션은 혼란이다. 상태가 살아남지 않으면 학습은 불가능하다.

Layer 2(오케스트레이션)와 Layer 3(기억)은 **상호 의존적**이다 — 타입된 핸드오프는 상태가 필요하고, 상태는 구조가 필요하다. 이 두 레이어는 함께 구축된다.

이 순서가 우선순위를 결정한다. 이전 철학 문서가 Planner-Worker 분리(C-1)를 Tier 1에 넣었던 것과 달리, **우리는 의도적으로 오케스트레이션을 뒤로 미룬다.** 에이전트가 무엇을 하는지 볼 수 있을 때까지, 오케스트레이션을 개선할 근거가 없다.

## Layer 0: 가시성 — 에이전트가 무엇을 하는지 본다

> 관찰할 수 없는 것은 디버그할 수 없고, 디버그할 수 없는 것은 신뢰할 수 없고, 신뢰할 수 없는 것은 프로덕션에 넣을 수 없다.

가시성은 "나중에 붙이는 것"이 아니라 **Day 1 아키텍처 결정**이다. 이것이 모든 것의 기반이다.

### 원칙: 에이전트는 자연어로 소통하고, 시스템이 모든 것을 기록한다

에이전트는 LLM이다 — 자연어가 본질이다. 에이전트 간 커뮤니케이션을 구조화된 JSON 이벤트로 강제하는 것은 에이전트의 본성에 역행한다.

대신:
- **에이전트는 자연어로 소통한다.**
- **시스템이 각 에이전트 턴을 자동으로 감싼다** — 누가, 누구에게, 언제, 어떤 태스크에 대해 — 구조화된 이벤트 로그 항목으로. 이것은 메타데이터 래핑이지, NLP 파싱이 아니다. 시스템은 커뮤니케이션의 *사실*을 기록하지, *의미적 내용*을 파싱하지 않는다.
- **이벤트 로그가 시스템의 진실의 원천(system of record)**이다 — 텔레메트리, 리플레이, 디버깅을 위해.

> 참고: 기존 하네스는 이미 JSONL 이벤트와 XML 기반 인텐트 블록을 시스템 레벨에서 기록한다. 이 원칙과 호환된다 — 에이전트 간 커뮤니케이션은 자연어, 시스템 레벨 텔레메트리는 구조화된 상태로 유지된다.

### 두 개의 관찰 표면

**1. 인간이 보는 창: 프로젝션 인터페이스**

에이전트의 자연어 커뮤니케이션과 이벤트 로그를 인간이 이미 사용하는 플랫폼(Slack, Discord, 커스텀 UI)에 **양방향으로 투사**한다.

- 인간은 에이전트의 작업을 실시간으로 **관찰**한다
- 인간은 메시지를 보내 **개입**한다 — 그 입력은 이벤트 로그에 기록되어 관련 에이전트에게 전달된다
- 인터페이스는 **교체 가능(pluggable)**하다 — Slack이 오늘의 권장 사항이지만, 내일은 커스텀 UI일 수 있다

Slack은 이미 에이전트-인간 상호작용의 지배적 채널이다 — Salesforce Agentforce, Dust.tt, 다수의 MCP 서버가 에이전트를 Slack 채널에 넣고 있다. "2026년에 멀티플레이어 AI가 싱글플레이어 AI를 잡아먹을 것이다" — 핵심은 AI를 팀이 이미 소통하는 곳으로 가져오는 것이다.

**2. 개발자가 파고드는 창: 텔레메트리 대시보드**

이벤트 로그를 기반으로 한 심층 분석 — 트레이싱, 비용 추적, 이상 탐지. 이것은 프로젝션 인터페이스와 분리된 개발자 도구다.

- OpenTelemetry 기반으로 벤더 lock-in을 방지 (IBM: 2026년 핵심 트렌드)
- 요청별, 사용자별, 기능별 비용 실시간 추적 (Braintrust: 세분화된 비용 분석)
- 에이전트의 "왜"를 로깅 — 행동뿐 아니라 추론 과정과 결정 경로 (Laminar: YC 지원, $3M 시드 — 모든 LLM 호출, 도구 사용, 브라우저 액션 캡처)
- 환각, 무한 루프, 비용 급증 자동 탐지 (Galileo: cascading failure 사전 식별)
- 전체 워크플로우의 조감도와 개별 에이전트의 상세 로그 모두 지원 (Claude HUD 패턴)
- 테스트/검증 환경을 인간에게도 에이전트에게도 동일하게 접근 가능하게

### 이 레이어의 요구사항

| ID | 요구사항 | 근거 |
|----|---------|------|
| V-1 | 모든 에이전트 행동을 이벤트 로그에 기록한다 (메타데이터 래핑). | 관찰 불가 = 디버그 불가 = 프로덕션 불가. |
| V-2 | 요청별/사용자별/기능별 비용을 실시간 추적한다. | Braintrust: 어떤 프롬프트/도구 호출이 비용을 올리는지 세분화 분석. |
| V-3 | 추론 과정과 결정 경로를 기록한다 ("왜"의 로깅). | 규제 감사 대비. o-mega.ai: "무엇뿐 아니라 왜를 로깅하라." |
| V-4 | 이상 행동(환각, 무한 루프, 비용 급증)을 자동 탐지하고 알린다. | Galileo: 환각 → 잘못된 도구 입력 → cascading failure. |

| V-6 | 전체 진행 상황 대시보드와 개별 에이전트 상세 로그를 모두 지원한다. | Claude HUD 패턴: 컨텍스트 사용량, 활성 도구, TODO 진행 상황. |

## Layer 1: 조율 — 에이전트가 서로 신호한다

> Layer 0 없이는 불가능하다: 에이전트 간 신호를 볼 수 없으면, 조율이 작동하는지 판단할 근거가 없다.

현재의 고통: 에이전트 팀은 단일 깊이에서만 협력한다. Agent A가 선행 작업을 끝내도 Agent B는 모른다. 상태가 유실된다.

### 원칙: 상태는 태스크 그래프에서 파생된다

에이전트의 상태 인식(done / blocked / need input)은 **태스크 그래프의 상태 변화**에서 파생된다 — `submitResult`, `claimReadyTasks` 같은 명시적 API를 통해. 에이전트의 자연어 출력을 NLP로 파싱하여 상태를 추출하지 않는다.

이 설계의 함의:
- **태스크 의존성 그래프**를 에이전트가 읽고 행동할 수 있어야 한다
- **계층적 협력**이 가능해야 한다 — 단일 깊이가 아니라 깊이 있는 협업
- **에이전트 수를 최소화**한다 — "더 많은 에이전트"가 아니라 "더 나은 프롬프트나 도구"로 해결할 수 있는지 먼저 확인 (Microsoft: "에이전트가 많아질수록 제어가 어려워진다")
- 병렬 실행 시 각 에이전트의 결과를 오케스트레이터가 **합성(synthesize)**하는 명시적 단계가 있어야 한다 (Lindy Agent Swarms)
- 에이전트 간 통신 프로토콜 표준화 — MCP(Anthropic)가 현재 가장 넓은 채택

### 이 레이어의 요구사항

| ID | 요구사항 | 근거 |
|----|---------|------|
| CO-1 | 에이전트 간 핸드오프는 타입이 지정되고 상태 계약이 명시적이어야 한다. | GitHub 엔지니어링: "에이전트를 분산 시스템의 컴포넌트처럼 취급하는 것이 신뢰성의 열쇠." |
| CO-2 | 에이전트 수를 최소화한다. 5~6인 이상의 그룹 챗은 혼란스럽다. | Microsoft Azure: "에이전트가 많아질수록 제어가 어려워진다." |
| CO-3 | 병렬 실행 결과를 합성하는 명시적 단계를 둔다. | Lindy Agent Swarms: 병렬 수행 후 결과 합성 필수. |
| CO-4 | 에이전트 간 통신 프로토콜을 표준화한다. | MCP가 현재 가장 넓은 채택. A2A(Google), UTCP 등 경쟁 중. |

## Layer 2: 오케스트레이션 — 계획하고, 배정하고, 검증한다

> Layer 0 + Layer 1 없이는 불가능하다: 에이전트를 볼 수 없고 조율할 수 없으면, 오케스트레이션은 단순히 혼란의 확장이다.
>
> Layer 3(기억)과 상호 의존적이다: 타입된 핸드오프는 상태가 필요하고, 상태는 구조가 필요하다. 두 레이어는 함께 구축된다.

### 원칙: Planner-Worker 분리, 그리고 작업 완료의 소유권

**Planner-Worker(계획자-실행자) 분리가 기본 아키텍처다.**

```
[인간] → 목표 제시
   ↓
[Planner/Orchestrator] — 고급 모델
   ├── 작업 분해
   ├── 위임 결정
   ├── 결과 검증
   └── 갈등 해소
   ↓        ↓        ↓
[Worker A] [Worker B] [Worker C] — 태스크에 따라 저렴한 모델
   ↓        ↓        ↓
[Reviewer/Checker] — 결과 교차 검증
   ↓
[Planner] ← 피드백 루프
```

이 패턴이 지배적인 이유: 계획에만 고급 모델을 쓰고 실행은 저렴한 모델로 돌림으로써 **비용 90% 절감** 가능 (Zylos Research). 오케스트레이터의 매 위임 결정이 로그 이벤트이므로 **디버깅 가능**. 하나의 Worker가 다운되면 오케스트레이터가 적응하는 **우아한 퇴화(graceful degradation)**.

**핵심 설계 결정: Worker는 태스크를 완료하지 않는다.**

Worker는 결과를 제출한다. **오케스트레이터가 태스크 완료를 소유한다.** 태스크를 done으로 마킹하기 전에, 오케스트레이터는 작업을 검증해야 한다 — 일반적으로 별도의 검증자/테스터 에이전트를 생성하여 출력을 검증한다. 이것이 오케스트레이션 레이어에서의 품질 게이팅이다.

**에이전트 타입은 사전 정의된 스키마다.** 사용자가 특정 도구 세트, 모델, 격리 설정으로 에이전트 타입을 정의한다. 오케스트레이터가 태스크당 적절한 에이전트 타입을 선택한다 — 에이전트가 동적으로 자기 자신을 설정하지 않는다.

| OR-9 | Shadow agent 탐지 메커니즘. | Okta: 비인가 에이전트 탐지를 핵심 기능으로 제시. |

## Layer 3: 기억 — 상태가 살아남는다

> Layer 0 없이는 불가능하다: 무엇을 기억할지 판단하려면 먼저 무엇이 일어나고 있는지 볼 수 있어야 한다.
>
> Layer 2(오케스트레이션)와 상호 의존적이다: 기억 없이는 타입된 핸드오프가 불가능하고, 오케스트레이션 없이는 기억의 구조가 무의미하다.

### 원칙: 기억은 큐레이션이다

인간 직원이 가치 있는 이유는 **경험이 축적**되기 때문이다. 첫 날의 인턴과 3년차 직원의 차이는 지능이 아니라 맥락(context)이다. 에이전트도 마찬가지다.

무엇을 기억할지, 무엇을 잊을지, 무엇을 요약할지의 판단이 에이전트 지능의 핵심 차원이다. Jason Liu가 말했듯, "compaction이 gradient descent의 momentum이라면, 언제 압축하느냐가 수렴 방향을 바꾼다."

### 세 층의 기억 아키텍처

**층 1: 작업 기억 (Working Memory) — RAM**
현재 작업의 컨텍스트 윈도우. 유한하고, 비싸고, 35분 이후 성능이 급격히 열화된다 (METR 연구: 작업 시간 2배 → 실패율 4배). 핵심은 "무엇을 넣을 것인가"가 아니라 **"무엇을 빼고도 동작하게 할 것인가"**이다.

프롬프트 엔지니어에게 **컨텍스트 품질은 최우선 가치**다. 모든 토큰은 수확 체감의 유한 자원이다. 낭비된 도구 호출, 실패한 시도, 중복된 탐색이 컨텍스트를 오염시키면 에이전트의 판단력이 저하된다. 컨텍스트 품질은 이 엔진의 1급 메트릭이다.

**층 2: 인덱스 기억 (Indexed Recall) — SSD**
시맨틱 검색, 지식 그래프(Cognee), 이벤트 기반 결정 로그(Mastra의 Observational Memory)를 통해 필요할 때 꺼내는 기억. 항상 벡터 기반 retrieval이 필요한 것은 아니다 — Anthropic과 OpenAI의 SKILL.md 시스템처럼 파일시스템 기반 목차에서 에이전트가 스스로 선택하는 패턴도 유효하다.

**층 3: 영속 기억 (Persistent Memory) — HDD**
세션을 넘어 유지되는 장기 기억. 핵심은 **에이전트가 스스로 "이건 기억해둬야 해"라고 판단하고 즉시 기록하는 것**이다. "나중에 compaction이 기억해주겠지"가 아니라 "지금 즉시 써둬라"가 정답이다 (OpenClaw 실무 패턴).

### 컴팩션 안전 재개 패킷 (Compaction-Safe Resume Packets)

Groundhog Day 문제의 구체적 해답. 기존 HarnessCheckpoint 스키마를 확장하여, 단순히 phase와 goal뿐 아니라 **도구 레지스트리 상태, 부분 결과, 의존성 그래프 버전**을 포함하는 타입된 상태 스냅샷을 생성한다. 컴팩션이 발생해도 에이전트는 정확히 중단된 지점에서 재개한다.

## Layer 4: 자기개선 — 관찰한 것으로부터 배운다

> Layer 0 없이는 불가능하다: 무엇이 일어났는지 볼 수 없으면 배울 수 없다.
> Layer 3 없이는 불가능하다: 배운 것을 유지할 수 없으면 개선이 아니라 반복이다.

정적인 에이전트는 그냥 자동화 스크립트다. 에이전트를 에이전트답게 만드는 것은 **실패로부터 배우는 능력**이다.

### 자기개선 루프

```
[실행] → [평가] → [반성] → [적응] → [실행] ...
```

1. **실행(Execution)**: 현재 로직으로 작업 수행.
2. **평가(Evaluation)**: LLM-as-Judge 또는 인간이 결과를 채점. 테스트 스위트 통과 여부. 사용자 만족도. 소요 시간.
3. **반성(Reflection)**: 메타-브레인(별도의 상위 LLM)이 실패의 근본 원인을 식별. OpenTelemetry 데이터를 정제하여 스스로 문제 원인을 채굴한다.
4. **적응(Adaptation)**: 프롬프트 수정, 새 도구 추가, 규칙 파일 업데이트, 성공 사례 저장.

**Ability.ai의 Claude Skills 패턴이 실용적 모델:** 에이전트가 여러 변형을 생성 → 인간이 최선을 선택 → `good_examples.md`에 저장. 실수 시 인간이 교정하면 에이전트가 자체 규칙 파일을 업데이트. 시간이 지나면 승인된 예시와 제약이 축적되어 에이전트가 비즈니스 표준에 수렴한다.

### 경계: 자기개선은 제한된다

**자기개선이 곧 자기 권한 확대가 되어서는 안 된다.** 프롬프트/규칙/도구 선택의 최적화는 허용하되, 자체 권한 확대나 보안 경계 수정은 금지한다. 자율 루프에 회로 차단기(circuit breaker)와 최대 반복 제한을 설정한다. 비수렴 시 인간에게 에스컬레이션한다.

에이전트는 자신의 능력 한계를 인식하고 **"이건 모르겠다"** 또는 **"이건 사람이 판단해야 한다"**고 선언할 수 있어야 한다. Inkeep이 말한 것: "Phase 간 게이트가 Phase 자체보다 중요하다."

### 이 레이어의 요구사항

| ID | 요구사항 | 근거 |
|----|---------|------|
| SI-1 | Evaluation 파이프라인을 CI/CD에 통합한다. | Braintrust: GitHub Actions 네이티브 통합. 매 PR마다 품질 게이팅. |
| SI-2 | LLM-as-Judge + 인간 전문가 평가의 하이브리드 체계를 구축한다. | OpenAI Self-Evolving Agents: 점수 임계값 초과 시 새 기본 에이전트로 교체. |

| SI-4 | 프로덕션 실패를 자동으로 evaluation 데이터셋으로 전환하는 폐루프를 구축한다. | Maxim AI: 프로덕션 → 평가 → 개선의 폐루프 아키텍처. |

## 모든 레이어를 관통하는 관심사

다음은 특정 레이어에 속하지 않지만 시스템 전체에 걸쳐 적용되는 관심사다.

### 자율성은 스펙트럼이다

Deloitte의 세 단계:

| 단계 | 이름 | 인간의 역할 | 에이전트의 역할 |
|------|------|------------|---------------|
| 1 | Human-in-the-loop | 매 결정에 승인 | 제안하고 대기 |
| 2 | Human-on-the-loop | 대시보드로 감시, 이상 시 개입 | 자율 실행하되 보고 |
| 3 | Human-out-of-the-loop | 결과만 확인 | 완전 자율 |

우리가 향하는 곳은 **단계 2: human-on-the-loop**이다. CoSAI 백서가 경고하는 **"consent fatigue"** — 사용자가 매번 뜨는 승인을 진지하게 읽으리라 가정하지 말라. 보안 제어는 승인 프롬프트가 아니라 **도구 자체에 내장**되어야 한다.

자율성의 수준은 (1) 작업의 위험도, (2) 에이전트의 검증된 신뢰도, (3) 되돌릴 수 있는가에 따라 동적으로 결정된다.

## 우선순위: 무엇을 먼저 만드는가

### Tier 1: 이것 없으면 시작할 수 없다

**Layer 0: 가시성** (핵심: 이벤트 로그 + CLI/대시보드. 스트레치: Slack/Discord 양방향 프로젝션)
**Layer 1: 조율** (태스크 그래프, 상태 신호, 계층적 협력)

이벤트 로그가 Tier 1의 핵심 결과물이다. 프로젝션 인터페이스(Slack/Discord)는 추가적이다.

> 이전 철학 문서가 C-1(Planner-Worker 분리)을 Tier 1에 넣었던 것과 달리, 우리는 의도적으로 오케스트레이션을 뒤로 미룬다. **에이전트가 무엇을 하는지 볼 수 있을 때까지, 오케스트레이션을 개선할 근거가 없다.**

### Tier 2: 이것 없으면 확장할 수 없다

**Layer 2 + Layer 3** (상호 의존적, 함께 구축)
- 오케스트레이션: Planner-Worker 분리, 교차 검증, 실행 환경
- 기억: 3-tier 아키텍처, 컴팩션 안전 재개 패킷, 컨텍스트 품질
- 비용 관리: 모델 라우팅, 예산 상한
- 기존 스택 통합: MCP, SKILL.md 패키징

### Tier 3: 이것 있으면 차별화된다

## 맺음

> **에이전트 시대의 승자는 가장 똑똑한 에이전트를 만드는 사람이 아니다.**
> **에이전트가 보이고, 조율되고, 기억하고, 스스로 개선되는 환경을 만드는 사람이다.**
>
> 그리고 그 순서가 중요하다. 볼 수 없으면 조율할 수 없고, 조율할 수 없으면 오케스트레이션은 혼란이고, 기억이 없으면 학습은 Groundhog Day다.
>
> 무얼 빌드하는지는 하네스를 사용하는 사람의 도메인 지식으로 출발한다. 이 엔진은 그 도메인 지식이 안전하게, 투명하게, 확장 가능하게 작동하는 기반을 만든다.
